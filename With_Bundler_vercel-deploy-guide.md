# 🚀 Node.js + TypeScript + Tsup + Vercel Deploy Guide

This guide sets up a TypeScript Node.js backend, bundled with **tsup**, and
deploys it to **Vercel**.

---

## 📦 1. Install Dependencies

```bash
npm i tsup
npm i -D tsx typescript
```

- `tsup` — bundles your TypeScript into `dist/`
- `tsx` — runs TypeScript directly for local dev (no compile step)
- `typescript` — needed for type-checking and for `tsx`/`tsup` to understand `.ts` files

---

## ⚙️ 2. Update `package.json`

```json
{
  "scripts": {
    "dev": "tsx watch ./src/server.ts",
    "build": "tsup",
    "start": "node dist/server.js",
    "typecheck": "tsc --noEmit"
  }
}
```

> ⚠️ **Fix from the original draft:** `build` must run **`tsup`**, not `tsc`.
> `tsc` doesn't know about `tsup.config.ts` and — without `outDir` set in
> `tsconfig.json` — won't emit anything into `dist/` at all. `tsup` is what
> actually produces `dist/server.js`. Use `tsc --noEmit` separately if you
> just want type-checking.

---

## ⚡ 3. Create `tsup.config.ts`

```ts
import { defineConfig } from "tsup";

export default defineConfig({
  entry: ["src/server.ts"],
  format: ["esm"], // Keep this as ESM
  target: "esnext",
  outDir: "dist",
  clean: true,
  bundle: true,
  splitting: false,
  sourcemap: true,

  // Add this banner to shim require() for CJS dependencies

  banner: {
    js: `
   import { createRequire } from 'module';
   const require = createRequire(import.meta.url);
  `,
  },
});

```

> ⚠️ **Fix from the original draft:** the banner must be scoped to the
> **esm** format only. CommonJS output doesn't understand `import` syntax —
> a static `banner.js` that's the same for both formats will inject an
> `import` statement into the **cjs** build and throw a syntax error the
> moment you run `node dist/server.js`.
>
> tsup's default extensions: `cjs` → `dist/server.js`, `esm` →
> `dist/server.mjs`. That's why `dist/server.js` is what `npm start` and
> `vercel.json` both point to below.

---

## 🧠 4. Update `tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "lib": ["ES2022"],
    "rootDir": "src",
    "outDir": "dist",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

> ⚠️ **Fix from the original draft:** `include`/`exclude` alone isn't a
> usable config — without `compilerOptions`, TypeScript defaults to very old
> settings (ES3 target, no `outDir`). This file is now used purely for
> type-checking and editor support; `tsup` does the actual bundling and
> ignores `outDir` here in favor of its own config.

---

## 🏗️ 5. Build the Project

```bash
npm run build
```

**Output:**

```
dist/server.js    # CommonJS — used by `npm start` and Vercel
dist/server.mjs    # ESM
dist/server.js.map
```

---

## 🌐 6. Install Vercel CLI

```bash
npm i -g vercel
```

## 🔐 7. Log in to Vercel

```bash
vercel login
```

## ⚙️ 8. Create `vercel.json`

Create this **before** deploying, so Vercel knows how to route requests:

```json
{
  "version": 2,
  "builds": [
    {
      "src": "dist/server.js",
      "use": "@vercel/node"
    }
  ],
  "routes": [
    {
      "src": "/(.*)",
      "dest": "dist/server.js"
    }
  ]
}
```

> 📝 **Note:** `builds`/`routes` is Vercel's legacy config format — it still
> works, but Vercel now recommends the `functions` property for new
> projects. If you want to modernize this later, the equivalent is roughly:
> ```json
> { "functions": { "dist/server.js": {} } }
> ```
> combined with a build command that runs `npm run build` first.

## 🚀 9. Deploy to Vercel

npm run build
npm start          # Test the built app
vercel             # Create a preview deployment
# Test the preview URL
vercel --prod      # Deploy to production if everything looks good

###For most professional teams, the workflow is:
-- first build the projet then test it with npm start or vercel dev-- then for preview deployment vercel then for production vercel --prod
- ✅ npm run build 
- ✅ npm start (or vercel dev)
- ✅ vercel (preview deployment)
- ✅ Verify the preview URL
- ✅ vercel --prod

```bash
vercel --prod
```

```
project-root/
│
├── src/
│   └── server.ts
│
├── dist/              ← generated by `npm run build`, don't commit
│   ├── server.js
│   └── server.mjs
│
├── package.json
├── tsconfig.json
├── tsup.config.ts

```

## ✅ Quick Checklist

- [ ] `npm run build` produces `dist/server.js` with no errors
- [ ] `npm start` runs `dist/server.js` locally without a require/import crash
- [ ] `vercel.json` points at `dist/server.js`
- [ ] `vercel --prod` deploys successfully
