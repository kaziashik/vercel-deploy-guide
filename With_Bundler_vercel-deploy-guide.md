# ЁЯЪА Node.js + TypeScript + Tsup + Vercel Deploy Guide

This guide sets up a TypeScript Node.js backend, bundled with **tsup**, and
deploys it to **Vercel**.

---

## ЁЯУж 1. Install Dependencies

```bash
npm i tsup
npm i -D tsx typescript
```

- `tsup` тАФ bundles your TypeScript into `dist/`
- `tsx` тАФ runs TypeScript directly for local dev (no compile step)
- `typescript` тАФ needed for type-checking and for `tsx`/`tsup` to understand `.ts` files

---

## тЪЩя╕П 2. Update `package.json`

```json
{
  "scripts": {
    "dev": "tsx watch ./src/server.ts",
    "build": "tsup",
    "start": "node dist/server.js",
    "typecheck": "tsc --noEmit"
  }
}
```

> тЪая╕П **Fix from the original draft:** `build` must run **`tsup`**, not `tsc`.
> `tsc` doesn't know about `tsup.config.ts` and тАФ without `outDir` set in
> `tsconfig.json` тАФ won't emit anything into `dist/` at all. `tsup` is what
> actually produces `dist/server.js`. Use `tsc --noEmit` separately if you
> just want type-checking.

---

## тЪб 3. Create `tsup.config.ts`

```ts
import { defineConfig } from "tsup";

export default defineConfig({
  entry: ["src/server.ts"],
  format: ["esm", "cjs"], // outputs dist/server.js (cjs) + dist/server.mjs (esm)
  target: "esnext",
  outDir: "dist",
  clean: true,
  bundle: true,
  splitting: false,
  sourcemap: true,

  // Fix require() compatibility тАФ ESM build only.
  banner: ({ format }) =>
    format === "esm"
      ? {
          js: `import { createRequire } from 'module'; const require = createRequire(import.meta.url);`,
        }
      : {},
});
```

> тЪая╕П **Fix from the original draft:** the banner must be scoped to the
> **esm** format only. CommonJS output doesn't understand `import` syntax тАФ
> a static `banner.js` that's the same for both formats will inject an
> `import` statement into the **cjs** build and throw a syntax error the
> moment you run `node dist/server.js`.
>
> tsup's default extensions: `cjs` тЖТ `dist/server.js`, `esm` тЖТ
> `dist/server.mjs`. That's why `dist/server.js` is what `npm start` and
> `vercel.json` both point to below.

---

## ЁЯза 4. Update `tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "lib": ["ES2022"],
    "rootDir": "src",
    "outDir": "dist",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

> тЪая╕П **Fix from the original draft:** `include`/`exclude` alone isn't a
> usable config тАФ without `compilerOptions`, TypeScript defaults to very old
> settings (ES3 target, no `outDir`). This file is now used purely for
> type-checking and editor support; `tsup` does the actual bundling and
> ignores `outDir` here in favor of its own config.

---

## ЁЯПЧя╕П 5. Build the Project

```bash
npm run build
```

**Output:**

```
dist/server.js    # CommonJS тАФ used by `npm start` and Vercel
dist/server.mjs    # ESM
dist/server.js.map
```

## ЁЯМР 6. Install Vercel CLI

```bash
npm i -g vercel
```

## ЁЯФР 7. Log in to Vercel

```bash
vercel login
```

## тЪЩя╕П 8. Create `vercel.json`

Create this **before** deploying, so Vercel knows how to route requests:

```json
{
  "version": 2,
  "builds": [
    {
      "src": "dist/server.js",
      "use": "@vercel/node"
    }
  ],
  "routes": [
    {
      "src": "/(.*)",
      "dest": "dist/server.js"
    }
  ]
}
```

> ЁЯУЭ **Note:** `builds`/`routes` is Vercel's legacy config format тАФ it still
> works, but Vercel now recommends the `functions` property for new
> projects. If you want to modernize this later, the equivalent is roughly:
> ```json
> { "functions": { "dist/server.js": {} } }
> ```
> combined with a build command that runs `npm run build` first.

## ЁЯЪА 9. Deploy to Vercel

npm run build
npm start          # Test the built app
vercel             # Create a preview deployment
# Test the preview URL
vercel --prod      # Deploy to production if everything looks good

###For most professional teams, the workflow is:
-- first build the projet then test it with npm start or vercel dev-- then for preview deployment vercel then for production vercel --prod
- тЬЕ npm run build 
- тЬЕ npm start (or vercel dev)
- тЬЕ vercel (preview deployment)
- тЬЕ Verify the preview URL
- тЬЕ vercel --prod

```bash
vercel --prod
```
