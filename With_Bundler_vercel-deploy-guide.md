# рЯЪА Node.js + TypeScript + Tsup + Vercel Deploy Guide

This guide sets up a TypeScript Node.js backend, bundled with **tsup**, and
deploys it to **Vercel**.

---

## рЯУ¶ 1. Install Dependencies

```bash
npm i tsup
npm i -D tsx typescript
```

- `tsup` вАФ bundles your TypeScript into `dist/`
- `tsx` вАФ runs TypeScript directly for local dev (no compile step)
- `typescript` вАФ needed for type-checking and for `tsx`/`tsup` to understand `.ts` files

---

## вЪЩпЄП 2. Update `package.json`

```json
{
  "scripts": {
    "dev": "tsx watch ./src/server.ts",
    "build": "tsup",
    "start": "node dist/server.js",
    "typecheck": "tsc --noEmit"
  }
}
```

> вЪ†пЄП **Fix from the original draft:** `build` must run **`tsup`**, not `tsc`.
> `tsc` doesn't know about `tsup.config.ts` and вАФ without `outDir` set in
> `tsconfig.json` вАФ won't emit anything into `dist/` at all. `tsup` is what
> actually produces `dist/server.js`. Use `tsc --noEmit` separately if you
> just want type-checking.

---

## вЪ° 3. Create `tsup.config.ts`

```ts
import { defineConfig } from "tsup";

export default defineConfig({
  entry: ["src/server.ts"],
  format: ["esm", "cjs"], // outputs dist/server.js (cjs) + dist/server.mjs (esm)
  target: "esnext",
  outDir: "dist",
  clean: true,
  bundle: true,
  splitting: false,
  sourcemap: true,

  // Fix require() compatibility вАФ ESM build only.
  banner: ({ format }) =>
    format === "esm"
      ? {
          js: `import { createRequire } from 'module'; const require = createRequire(import.meta.url);`,
        }
      : {},
});
```

> вЪ†пЄП **Fix from the original draft:** the banner must be scoped to the
> **esm** format only. CommonJS output doesn't understand `import` syntax вАФ
> a static `banner.js` that's the same for both formats will inject an
> `import` statement into the **cjs** build and throw a syntax error the
> moment you run `node dist/server.js`.
>
> tsup's default extensions: `cjs` вЖТ `dist/server.js`, `esm` вЖТ
> `dist/server.mjs`. That's why `dist/server.js` is what `npm start` and
> `vercel.json` both point to below.

---

## рЯІ† 4. Update `tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "lib": ["ES2022"],
    "rootDir": "src",
    "outDir": "dist",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

> вЪ†пЄП **Fix from the original draft:** `include`/`exclude` alone isn't a
> usable config вАФ without `compilerOptions`, TypeScript defaults to very old
> settings (ES3 target, no `outDir`). This file is now used purely for
> type-checking and editor support; `tsup` does the actual bundling and
> ignores `outDir` here in favor of its own config.

---

## рЯПЧпЄП 5. Build the Project

```bash
npm run build
```

**Output:**

```
dist/server.js    # CommonJS вАФ used by `npm start` and Vercel
dist/server.mjs    # ESM
dist/server.js.map
```

---

## рЯМР 6. Install Vercel CLI

```bash
npm i -g vercel
```

## рЯФР 7. Log in to Vercel

```bash
vercel login
```

## вЪЩпЄП 8. Create `vercel.json`

Create this **before** deploying, so Vercel knows how to route requests:

```json
{
  "version": 2,
  "builds": [
    {
      "src": "dist/server.js",
      "use": "@vercel/node"
    }
  ],
  "routes": [
    {
      "src": "/(.*)",
      "dest": "dist/server.js"
    }
  ]
}
```

> рЯУЭ **Note:** `builds`/`routes` is Vercel's legacy config format вАФ it still
> works, but Vercel now recommends the `functions` property for new
> projects. If you want to modernize this later, the equivalent is roughly:
> ```json
> { "functions": { "dist/server.js": {} } }
> ```
> combined with a build command that runs `npm run build` first.

## рЯЪА 9. Deploy to Vercel

```bash
vercel --prod
```
